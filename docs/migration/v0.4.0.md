# Migration Guide: v0.3.0 to v0.4.0

This guide covers all changes since the 3/23/2026 baseline (commit `82988ef`). If you are migrating from v0.2.0 or earlier, follow the [v0.3.0 migration guide](v0.3.0.md) first.

## Summary of Changes

v0.4.0 is a major update to the dashboard, pipeline, and documentation. The flat traceability report has been replaced with a 7-tab interactive dashboard, a global release filter, SBOM/vulnerability integration, and a centralized filter architecture.

---

## Breaking Changes

### 1. Data Model: "Verification Method" renamed to "Verification Criteria"

This was introduced in v0.3.0 but is listed here since the 3/23 baseline predates it.

| Before | After |
|--------|-------|
| `verificationMethods` (JSON array) | `verificationCriteria` |
| `verificationMethodId` | `verificationCriteriaId` |
| `SYS-REQ-001-VM-01` (scoped ID) | `SYS-REQ-001-VC-01` |
| `@VM:SYS-REQ-001-VM-01` (Gherkin tag) | `@VC:SYS-REQ-001-VC-01` |
| `class VerificationMethod` (Python) | `class VerificationCriteria` |

**Backward compatibility**: The Python code still accepts old JSON field names (`verificationMethods`, `verificationMethodId`) as fallbacks. However, **Gherkin tags have no backward compatibility** -- `@VM:` tags will not be recognized. All feature files must use `@VC:` tags.

### 2. Dashboard Output File

| Before | After |
|--------|-------|
| `traceability_report.html` (flat table, 344 lines) | `MBSE_Traceability_Dashboard.html` (7-tab interactive, ~2000 lines) |

The HTML output is a completely different format. Any tooling that parses the old HTML will break. The JSON output (`final_traceability_report.json`) retains the same schema with additional fields.

### 3. Report Generator CLI

New optional flags added:

```bash
python tools/report_generator.py \
  --requirements build/cameo/requirements/requirements.json \
  --behave-results build/reports/bdd/behave-results.json \
  --traceability-input build/reports/traceability/traceability_report.json \
  --output-json build/reports/traceability/final_report.json \
  --output-html build/reports/traceability/MBSE_Traceability_Dashboard.html \
  --release-plan release-plan.json \         # NEW: enables release filter
  --sbom-path build/reports/security/sbom.json \      # NEW: Cyber tab
  --grype-path build/reports/security/grype-results.json  # NEW: Cyber tab
```

All new flags are optional. Omitting them produces the dashboard with graceful empty states.

### 4. Traceability Checker CLI

New optional flags:

```bash
python tools/traceability_checker.py \
  ...existing flags... \
  --release-plan release-plan.json \  # NEW: enables deferred VCs
  --release 1.0.0                     # NEW: override VERSION file
```

### 5. Pipeline Workflow

The `traceability-report` job has been renamed to `traceability-dashboard`. The output artifact is now named `dashboard` (in addition to `all-reports`).

## New Features

### 7-Tab Interactive Dashboard

The flat HTML report has been replaced with a self-contained interactive dashboard:

| Tab | Content |
|-----|---------|
| Executive Summary | Readiness ring, release progress, top issues, cyber risk summary |
| Traceability Matrix | Accordion drill-down: Requirement -> VCs -> Scenarios with steps |
| Quality Gates | Gate A (Coverage), Gate B (Drift), Gate C (Orphans) with clickable items |
| Release Progress | Per-release cards with progress bars and clickable VC lists |
| Test Execution | Behave results: features -> scenarios -> steps with keyword highlighting |
| Cyber | SBOM inventory + vulnerability table with severity filtering and CSV export |
| Export & Info | Print, JSON download, metadata, legend, glossary |

Features:
- Self-contained HTML with zero external dependencies (air-gapped compatible)
- Cache-busting meta tags prevent stale browser caching
- XSS protection via `<\/script>` escaping in JSON injection
- `@media print` support
- Cross-tab navigation with yellow flash highlighting

### Global Release Filter

A dropdown in the hero header scopes the entire dashboard to a specific release version. All tabs re-render through a centralized `buildSnapshot()` architecture.

- "All Releases" shows everything (default)
- Released versions show all gates passed, no orphans, no vulnerabilities, shipped confirmation
- Scope is cumulative -- selecting 1.1.0 includes 1.0.0 + 1.1.0 VCs

### Release Planning System

A `release-plan.json` file maps requirements/VCs to release versions:

```json
{
  "currentVersion": "1.1.0",
  "releases": [
    {
      "version": "1.0.0",
      "targetDate": "2026-06-01",
      "description": "Initial deployment",
      "scope": ["SYS-REQ-001", "SYS-REQ-002"]
    }
  ]
}
```

- Scope supports requirement-level (`SYS-REQ-001`) and VC-level (`SYS-REQ-001-VC-02`)
- Out-of-scope VCs shown as "deferred" with target release
- Coverage % calculated against in-scope VCs only
- See [docs/guides/release-management.md](../guides/release-management.md) for the full lifecycle guide

### SBOM and Vulnerability Integration

The Cyber tab displays data from Syft (SBOM) and Grype (vulnerability scanner):
- Component inventory with type breakdown
- Vulnerability table with severity filtering, CVE search, and CSV export
- Policy compliance banner (PASS/WARNING/FAIL based on severity thresholds)
- Fix availability summary

### Hero Header

Three clickable cards above the tabs:
- **Release Scope** -- current release, VC count, deferred/manual counts
- **Test Results** -- pass/fail with rate bar and failed VC list
- **Blockers** -- count with category breakdown

Full coverage bar showing all VC statuses proportionally.

### Demo Data Generator

```bash
python tools/generate_demo_data.py --output-dir build/demo
```

Generates synthetic data exercising every dashboard state: pass, fail, uncovered, drifted, manual, orphaned, deferred. Includes SBOM (12 components), Grype (7 CVEs), and a 3-release plan.

### Test Suite

232 pytest tests covering:
- Report generation logic (43 base tests)
- Demo dashboard end-to-end validation (68 tests)
- SBOM/Grype integration tests (121 tests via TestDemoWithSBOM)

## Migration Steps

### Step 1: Update your Gherkin tags

Replace all `@VM:` tags with `@VC:`:

```bash
# Find files with old tags
grep -rn "@VM:" bdd/features/

# Replace (review each change)
find bdd/features -name "*.feature" -exec sed -i 's/@VM:/@VC:/g' {} +
```

### Step 2: Update your requirements JSON (if needed)

If your Cameo export macro still produces `verificationMethods` / `verificationMethodId`, the Python code will accept it. However, update the macro to use the new field names when convenient:

- `verificationMethods` -> `verificationCriteria`
- `verificationMethodId` -> `verificationCriteriaId`

### Step 3: Create a release plan

Create `release-plan.json` in the repository root. See [docs/guides/release-management.md](../guides/release-management.md) for the full schema and lifecycle guide.

If you don't create one, the dashboard works without it -- the Release Progress tab shows an empty state, and all VCs are treated as in-scope.

### Step 4: Update your pipeline

If using GitHub Actions, the workflow now expects `generate_demo_data.py` for the prototype. When connecting real data, update the `traceability-dashboard` job to point at real paths (documented in comments in the workflow file).

If using Jenkins, update the report generator invocation to pass the new optional flags:

```bash
python tools/report_generator.py \
  ...existing flags... \
  --release-plan release-plan.json \
  --sbom-path build/reports/security/sbom.json \
  --grype-path build/reports/security/grype-results.json
```

### Step 5: Update the traceability baseline

After updating tags and requirements, refresh the baseline:

```bash
python tools/traceability_checker.py \
  --requirements build/cameo/requirements/requirements.json \
  --features-dir bdd/features \
  --stubs-output-dir bdd/features/automated \
  --non-test-output-dir bdd/features/non_test \
  --report-output /dev/null \
  --update-baseline
```

### Step 6: Install dependencies

```bash
pip install -r tools/requirements.txt
```

No new dependencies were added in this release.

### Step 7: Run tests

```bash
python -m pytest tests/ -v
```

All 232 tests should pass.
